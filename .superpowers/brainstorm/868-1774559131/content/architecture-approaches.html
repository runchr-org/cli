TUI Architecture Approaches

Three ways to structure the bubbletea dashboard

A

Single-File Flat Model

All TUI logic in one memory_loop_tui.go file within the cli package. One big Model struct with a switch on active tab index.

Pros

  • No new packages
  • Follows existing single-file pattern

Cons

  • 1500+ lines, untestable per-tab
  • Merge conflict magnet
  • Violates Go focused-file convention
B

Dedicated memorylooptui/ Package (Recommended)

New package with root model + per-tab sub-models in separate files. Root owns state, sub-models communicate via messages. Standard Charm ecosystem pattern.

Pros

  • Each tab testable in isolation
  • Clean separation of concerns
  • Canonical bubbletea architecture
  • Matches existing package-per-feature style

Cons

  • More files to navigate
  • Need message-passing discipline
memorylooptui/
  root.go          // Root model, tab switching, program entry
  tab_memories.go  // Table + detail + lifecycle actions
  tab_injection.go // Log table + prompt tester
  tab_history.go   // Refresh history + async trigger
  tab_settings.go  // Mode/policy/max cycling
  keys.go          // Keybinding definitions
  styles.go        // Lipgloss styles from termstyle
  messages.go      // Custom tea.Msg types
  render.go        // Shared helpers (tab bar, status bar)
C

Embed Inside huh Forms

Build the TUI as chained huh forms with custom field types. Reuse NewAccessibleForm pattern. Tab switching simulated by form chaining.

Pros

  • No new direct dependencies
  • ACCESSIBLE mode comes free

Cons

  • huh designed for linear forms, not dashboards
  • No table component — must rebuild bubbles inside huh
  • Loses scroll position between "tabs"
  • Fighting the abstraction
